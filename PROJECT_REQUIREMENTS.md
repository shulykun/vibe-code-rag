## Модуль RAG по исходному коду Java‑проектов (MCP‑tool)

### 1. Цель и позиционирование

- **Назначение**: модуль Retrieval‑Augmented Generation (RAG) для глубокого понимания и навигации по исходному коду Java‑проектов (monolith, microservices, multi‑module Maven/Gradle, monorepo).
- **Основное применение**: системы vibe‑coding и ассистенты разработчика для больших Java‑кодовых баз.
- **Форм‑фактор**: MCP‑сервер (tool), к которому IDE/клиенты подключаются по JSON‑RPC.
- **Язык реализации MCP‑сервера**: **Python**.

### 2. Ключевые UX‑фичи (что должен уметь помощник)

- **Семантические чанки кода**
  - Чанкинг не по строкам, а по **методам/классам**, с учётом:
    - сигнатуры, тела метода;
    - Javadoc и значимых комментариев;
    - принадлежности к модулю/слою (web/service/repository и т.п.).

- **Граф зависимостей**
  - Понимание связей вида: `UserService → UserRepository → JpaUserEntity`.
  - Возможность отвечать на вопросы:
    - “кто использует этот класс/интерфейс?”;
    - “какие сервисы зависят от этого репозитория?”;
    - “что сломается, если изменить этот метод?” (выдавать топ‑N кандидатов).

- **Контекстные подсказки для vibe‑coding**
  - На вход приходит контекст из IDE (путь файла, позиция курсора, выделенный текст, ветка).
  - Инструмент возвращает:
    - объяснение текущего куска кода;
    - примеры использования похожих методов/классов;
    - предложения по изменению с указанием затронутых мест (“вот 3 места, которые сломаются”).

- **Интерактивный поиск**
  - Возможность уточнять запросы:
    - “найди ещё, но только в тестах”;
    - “покажи только REST‑контроллеры”;
    - “ограничься модулем `user-service`”.
  - Поддержка фильтров по модулю, пакету, типу артефакта (класс, тест, конфиг, документация).

### 3. Архитектура (логические компоненты)

- **IDE / клиенты**
  - Cursor, Continue, Claude Desktop и др., отправляют JSON‑RPC запросы к MCP‑серверу.

- **MCP‑сервер (Python)**
  - Экспортирует инструменты MCP (например, `index_project`, `project_query`, `editor_context_query`).
  - Инкапсулирует всю работу по индексации, поиску и RAG‑оркестрации.

- **Индексатор кода**
  - Читает структуру проекта (Maven/Gradle, multi‑module).
  - Парсит Java‑код через **tree‑sitter** (или другую AST‑библиотеку) с построением:
    - семантических чанков (методы, классы, конфиги);
    - графа зависимостей (класс→класс, модуль→модуль).

- **Семантический поиск и ранжирование**
  - Векторный поиск по чанкам кода и документации.
  - Классический поиск по символам/именам (классы, методы, пакеты).
  - Реранк результатов (по желанию — через cross‑encoder).

- **Vector DB**
  - Qdrant / Chroma / другой стор с поддержкой фильтрации по метаданным (модуль, пакет, тип артефакта).

- **Java‑проект**
  - Исходники, ресурсы, тесты, документация, конфиги (источник истины для индексации).

### 4. Функциональные требования (по модулям)

- **Модуль `ProjectScanner`**
  - Определение типа сборки (Maven/Gradle).
  - Обнаружение модулей, исходников (`src/main/java`, `src/test/java` и др.), ресурсов (`resources/`), конфигов.
  - Поддержка monorepo и многомодульных проектов (модуль → отдельный namespace в индексе).

- **Модуль `CodeParser`**
  - Парсинг Java‑кода через AST (tree‑sitter или аналог).
  - Извлечение:
    - классов, интерфейсов, enum’ов;
    - методов, полей, сигнатур;
    - импортов и зависимостей;
    - аннотаций (Spring, JPA, тестовые фреймворки и др.);
    - комментариев и Javadoc.

- **Модуль `Chunker`**
  - Формирование семантических чанков:
    - уровне методов и классов (код + Javadoc + важные комментарии);
    - уровни конфигурации (блоки в `application.yml`, `pom.xml`, `build.gradle` и т.п.);
    - документация (README, ADR, `docs/` и др.).
  - Учёт ограничений по размеру чанка (токены/символы) под выбранную LLM.

- **Модуль `DependencyGraph`**
  - Построение графа зависимостей:
    - вызовы методов, использование классов/интерфейсов;
    - связи между слоями и модулями.
  - API для запросов:
    - “входящие/исходящие зависимости для класса/метода/модуля”;
    - “потенциальные места поломки” при изменении сущности.

- **Модуль `EmbeddingStore`**
  - Построение эмбеддингов для:
    - чанков кода;
    - документации и комментариев;
    - метаданных (имя класса/метода, путь файла, модуль, тип артефакта).
  - Хранение и поиск:
    - поддержка векторного стора (по умолчанию Qdrant, опционально Chroma/FAISS);
    - операции: добавление/обновление чанков, поиск top‑K по запросу;
    - фильтрация по модулю, пакету, типу артефакта.

- **Модуль `Retriever`**
  - Обработка пользовательского запроса:
    - нормализация текста запроса;
    - поиск по ключевым словам / символам (по имени класса/метода/пакета);
    - семантический поиск по эмбеддингам;
    - объединение, ранжирование и дедупликация результатов;
    - поддержка интерактивных фильтров (модуль, слой, тесты/прод‑код).

- **Модуль `RagOrchestrator`**
  - Формирование промпта для LLM на основе результатов `Retriever` и графа зависимостей.
  - Вызов LLM‑провайдера (через конфигурируемый API‑клиент).
  - Пост‑обработка ответа:
    - добавление ссылок на файлы, классы, методы, модули;
    - компактное представление результата для IDE/клиента.

- **Интеграционный слой (MCP +, опционально, HTTP/gRPC)**
  - **MCP‑инструменты (минимум)**:
    - `index_project` — индексация проекта или выбранных модулей/подкаталогов;
    - `project_query` — общий запрос по коду/архитектуре с optional‑фильтрами (модуль, пакет, слой, “только тесты”);
    - `editor_context_query` — запрос с учётом контекста редактирования (файл, курсор, выделение).
  - **Опциональный HTTP/gRPC API**:
    - `/indexProject`, `/query`, `/editorSuggest` — для использования вне MCP‑клиентов.

### 5. Технологический стек (рекомендация)

- **MCP‑сервер**: Python.
- **Парсер/чанкинг**: tree‑sitter для Java (через Python‑биндинги или отдельный сервис).
- **Vector DB**: Qdrant (фильтрация по метаданным, шардирование), альтернатива — Chroma/FAISS для локального режима.
- **Эмбеддинги**:
  - локально: `jina-embeddings-v2-base-code` (или аналогичный кодовый эмбеддер);
  - через API: Voyage Code / иные специализированные кодовые эмбеддинги.
- **Реранкинг (опционально)**: cross‑encoder для rerank top‑K результатов перед подачей в LLM.

### 6. Нефункциональные требования

- **Производительность**
  - Поддержка больших кодовых баз (десятки и сотни тысяч Java‑файлов).
  - Параллельное сканирование и парсинг (где возможно).
  - Инкрементальная переиндексация (индексация только изменённых файлов/модулей).
  - Возможность частичной загрузки индекса (lazy loading) для отдельных модулей/подсистем.

- **Масштабируемость и расширяемость**
  - Возможность смены векторного стора без изменения внешнего API.
  - Возможность конфигурируемого выбора LLM‑провайдера.
  - Возможность добавить поддержку других JVM‑языков (Kotlin, Scala) без переработки архитектуры.
  - Поддержка распределённого/кластерного векторного стора с шардированием по модулям или директориям.

- **Надёжность**
  - Устойчивость к частично некорректному коду (недоскомпилированным веткам, конфликтам).
  - Логирование процесса индексации и запросов.

- **Развёртывание**
  - Python‑пакет, который можно подключить как библиотеку.
  - Возможность запуска как standalone‑сервиса (Docker‑контейнер).
  - Конфигурация через файл/переменные окружения (настройка LLM, стора, путей к проекту, опций шардирования).

### 7. Минимальный полезный функционал (MVP)

- Индексация Java‑проекта (структура + основные классы/методы) с семантическими чанками.
- Построение базового графа зависимостей (классы/методы, без глубокого анализа фреймворков).
- Семантический поиск по коду и README/документации.
- Базовый RAG‑pipeline:
  - поиск top‑K фрагментов;
  - формирование промпта;
  - вызов LLM и возврат ответа.
- Экспорт функционала как MCP‑tool с базовым набором инструментов:
  - `index_project`;
  - `project_query`.

### 8. Дальнейшее развитие

- Более детальная навигация по архитектуре (с учётом фреймворков, слоёв, bounded contexts).
- Специализированные сценарии для Spring, JPA, тестовых фреймворков.
- Умные подсказки “что сломается” на основе графа зависимостей и истории изменений.
- Более тесная интеграция с IDE (контекст коммита, ветки, ревью, статический анализ).
